# QuickDep 问题定义与修复计划

## 1. 这份文档要解决什么

这份文档不是实验摘要，也不是泛泛而谈的反思笔记。

它要回答 4 个实际问题：

1. QuickDep 这次到底暴露了什么真正问题
2. 哪些是产品边界问题，哪些是实现问题
3. 后续应该先修什么，后修什么
4. 修到什么程度，才算这轮问题真正被收敛

如果这份文档不能直接指导任务排序和验收，它就还不够好。

## 2. 真正的问题是什么

这次暴露出来的核心问题不是“某个语言支持不完整”，也不是“某次实验误报率高”本身。

真正的问题是：

> 我们把一个“静态结构定位工具”，拿去回答一个“删除安全性判断问题”。

QuickDep 当前天然擅长的是：

- 找相关文件
- 找相关符号
- 找静态调用关系
- 帮 agent 缩小阅读范围

但死代码清理真正要判断的是：

- 这个符号能不能删
- 删了会不会在运行时出问题
- 有没有动态注册、反射、事件、框架约定、配置驱动等隐式引用

因此，`incoming = 0` 只能说明：

- 当前静态图里没有发现调用者

不能直接说明：

- 这个符号可以删除
- 这个符号一定是死代码

这就是当前产品体验和用户预期之间最根本的错位。

## 3. 哪些是边界问题，哪些是实现问题

### 3.1 产品边界问题

这些问题不是“多补几个 parser”就能解决的：

- 把“候选”说成了“判决”
- 把“静态证据不足”说成了“可安全删除”
- 没有向 agent 明确表达结果的不确定性
- 没有把 QuickDep 放回“先缩小范围，再继续验证”的正确位置

### 3.2 实现问题

这些问题应该直接进入开发清单：

- 动态模式没有风险标记
- 前端框架约定式调用支持弱
- 输出里缺少符号角色、验证建议、风险等级
- MCP 工具对 agent 的输入不够宽容
- 项目状态持久化不够自愈，容易被旧状态污染

边界问题决定产品怎么说，实现在此基础上决定工具怎么做。

## 4. 当前版本真正有价值的地方

这次实验并没有证明 QuickDep “没用”，相反，它证明了 QuickDep 在正确问题上是有价值的。

QuickDep 现在真正成立的价值是：

- 在大型项目里快速给出结构化候选范围
- 让 agent 不必先盲目 grep 和盲目读文件
- 让“先看哪里”这件事更快、更稳
- 给人工验证和后续工具调用提供一个高信息密度的入口

一句话说，QuickDep 现在更像：

> 结构化缩圈器、复核入口、依赖定位助手

而不是：

> 删除安全性的独立裁判

## 5. 为什么这次会失真

### 5.1 问题定义错位

我们用“静态边不存在”替代了“删除安全性证明”。

但删除安全性至少还需要：

- 文本引用证据
- 动态调用证据
- 框架约定入口证据
- 公开 API / 内部实现 / 扩展点等角色证据
- 编译与测试结果

### 5.2 证据模型太单薄

QuickDep 当前主要只有一类强证据：

- 静态依赖边

而真实项目里经常还有这些不会稳定落到静态边上的调用模式：

- Python 字典注册
- Vue / 前端动态导入
- Pinia Store 工厂
- 事件订阅和广播
- 字符串驱动分发
- 反射和约定式框架入口

### 5.3 输出语义不够诚实

当前结果很容易让人把：

- `incoming = 0`

理解成：

- `dead`

但更诚实的表达应该是：

- `unused_candidate`
- `no_static_callers_found`
- `dynamic_risk_unknown`

也就是说，当前工具输出的是“缺乏静态证据”，但呈现出来像“最终判决”。

## 6. 文档本身还缺什么

如果只停在“问题定义错位”这个结论，这份文档仍然不够。

它还缺 3 件事：

1. 缺修复顺序
   现在只是指出问题，没有明确 P0 / P1 / P2 的先后级。

2. 缺实现落点
   现在没有明确哪些属于 README / 输出 schema / MCP 接口 / 动态风险检测。

3. 缺验收标准
   现在没有定义“修好了是什么样”，后续容易反复空转。

所以这份文档必须从“诊断”升级为“修复计划”。

## 7. 修复原则

这轮修复不应该从“再补更多语言细节”开始，而应该遵循下面的顺序：

1. 先修语义误导
2. 再修 agent 易用性
3. 再补风险分层
4. 最后再做框架和动态模式增强

原因很简单：

- 如果语义还是错的，能力越强越容易被误用
- 如果 agent 接口本身就容易踩坑，再好的分析能力也用不稳
- 如果没有风险分层，新增解析能力也很难转化成正确决策

## 8. 分阶段修复计划

### P0：修正输出语义和调用体验

目标：

- 不再让结果看起来像“死代码判决”
- 不再让 agent 因参数格式和脏状态轻易失败

要做的事：

- 把“死代码候选”统一改成“未发现静态调用者候选”
- 在输出里显式区分 `candidate` 和 `confirmed`
- MCP 工具兼容更自然的 project 输入形态
- 清理或跳过 manifest 中已经失效的项目状态
- 在错误信息里明确告诉 agent 应该如何补充参数或重试

本轮已经开始触达的点：

- `ProjectTarget` 兼容字符串路径输入
- `list_projects` 遇到失效项目时不再整体失败

验收标准：

- agent 直接传路径字符串时，`scan_project` / `get_scan_status` / `locate_relevant_code` 不再失败
- 一个失效项目状态不会拖垮整次 `list_projects`
- README 和工具文案里不再把 `incoming = 0` 暗示成“可删”

### P1：把结果从“结论”改成“证据包”

目标：

- 让 agent 知道“为什么怀疑这个符号”
- 让 agent 知道“下一步应该怎么验证”

要做的事：

- 为候选结果增加结构化字段
- 至少补齐下面这些信息：

| 字段 | 作用 |
| --- | --- |
| `static_incoming_count` | 当前静态入边数量 |
| `exported` | 是否对外导出 |
| `symbol_role` | 公开 API / 内部实现 / 测试专用等 |
| `dynamic_risk` | 低 / 中 / 高 |
| `verification_hints` | 建议继续 grep、读哪些文件、查哪些注册点 |
| `confidence` | 当前判断可信度 |

验收标准：

- agent 不再只能看到 `incoming = 0`
- agent 可以直接根据输出继续完成二次验证
- 输出默认语义从“结论”变成“证据 + 建议”

### P2：补动态风险识别，而不是先追求完美解析

目标：

- 先知道哪里“不该轻信”，再追求哪里“能完全推断”

第一批值得优先补的模式：

- Python `HANDLERS = { "...": func }`
- Python `COMMANDS`, `ROUTES`, `REGISTRY` 一类函数表
- Vue `defineAsyncComponent(() => import(...))`
- Pinia `defineStore(...)`
- 常见事件订阅 / 发布模式

这里的目标不是 100% 推断所有真实调用，而是把这些符号至少标成：

- `dynamic_entry_candidate`
- `needs_text_verification`

验收标准：

- 动态模式命中的符号不再默认给出低风险结果
- agent 能收到“这里存在动态调用风险”的显式提示

### P3：形成一条可落地的“安全删除工作流”

目标：

- 让 QuickDep 成为删除验证链条的一环，而不是一个孤立判断器

建议工作流：

1. QuickDep 给候选范围
2. QuickDep 给风险和验证建议
3. grep / 文本搜索补静态图之外的证据
4. 人工确认设计意图和边界
5. 编译 / 测试验证删除安全性

可能新增的高阶工具形态：

- `get_removal_candidates`
- `get_verification_context`
- `get_refactoring_context`

验收标准：

- agent 能从 QuickDep 输出自然进入下一步验证
- 删除建议默认是“请验证”，而不是“请删除”

## 9. 暂时不要做什么

在上面这些基础问题没有收敛前，不建议优先做这些事：

- 继续把“死代码识别”包装成主要卖点
- 只追求误报率数字，而不改结果语义
- 一次性补完所有语言的所有动态模式
- 在没有风险分层的前提下输出“可删除清单”

这些事会继续放大误用风险。

## 10. 修好之后应该是什么样

这轮问题如果真的修好了，应该看到的是下面这些变化。

### 对 agent

- agent 更容易把工具调起来，不会频繁踩参数格式坑
- agent 拿到的是“证据包”，不是一句模糊结论
- agent 知道什么时候该信，什么时候该继续验证

### 对工程师

- 工程师看到的结果不再像“误导性判决”
- 工程师可以把 QuickDep 当作候选生成器和结构化入口
- 工程师更容易把它接进删除、重构、排查这类真实工作流

### 对产品口径

- QuickDep 的主故事回到“大项目里的快速定位与缩圈”
- 死代码场景被表述为“候选发现 + 风险提示 + 验证辅助”
- 对外宣传和实际能力边界重新对齐

## 11. 一个更准确的产品表述

当前更准确的对外描述应该是：

> QuickDep 帮助 agent 和工程师在大型项目里先缩小范围、找到值得读的依赖关系，再结合文本搜索、编译、测试和人工判断完成最终决策。

如果聚焦到死代码场景，则应该表述为：

> QuickDep 负责生成“未发现静态调用者”的候选清单，标注风险和验证方向，而不是直接判定代码可删除。

## 12. 修复完成后如何替换这份文档

这份文档的定位是：

- 修复前的问题定义
- 修复中的任务排序依据
- 修复验收时的对照基线

它不应该长期作为面向普通用户的稳定说明文档存在。

当 `P0` 到 `P3` 的核心内容完成后，这份文档应该被新的“现状文档”替换，新的文档应当改写为：

- QuickDep 当前已经解决什么
- QuickDep 当前仍然不解决什么
- agent 应该如何正确使用它
- 删除、重构、定位这几类场景下的推荐工作流

也就是说，修复完成之后，当前内容要从：

- “问题和计划”

替换为：

- “修复后的真实边界和使用方式”

建议到时直接重写本文件，而不是继续无限追加历史问题。

重写后的版本至少应该包含：

1. 当前能力边界
2. 当前输出语义
3. 推荐工作流
4. 高风险场景提醒
5. 与 grep / 编译 / 测试的关系

这样可以避免仓库里长期保留一份已经过时的问题诊断，误导后续开发和用户理解。

## 13. 结论

这次真正暴露的问题，不是“工具没价值”，而是：

> 问题定义、输出语义、agent 调用体验，没有对齐真实使用场景。

因此这轮修复的重点不是继续堆 parser，而是先完成下面三件事：

1. 把结果从“判决语气”改成“证据语气”
2. 把接口从“人类知道怎么传”改成“agent 不容易传错”
3. 把 QuickDep 放回“缩小范围和复核入口”的正确位置

这三件事如果做对了，QuickDep 的能力边界会更清晰，agent 的使用体验会更稳，后续再补语言和框架支持也更有方向。
